# 架构决策记录

## 1. 为什么 ReAct 而不是 Plan-Execute？

ReAct 的 Thought → Action → Observation 循环适合工具数量适中（<15个）、单次请求工具调用次数少（2-5次）的场景。本项目工具间依赖不强，Plan-Execute 的 DAG 复杂度收益不明显。

## 2. 为什么双层记忆？

- SQLite：结构化存储（画像/对话），支持精确查询和持久化
- Chroma：向量存储，支持语义检索（"上次说的那个..."→召回相关对话）

单用 SQLite 无法做语义检索，单用 Chroma 无法做精确过滤。

## 3. 旅行规划为什么用规则筛选而不是 RAG？

高德 API 返回的 POI 数量有限（~60条），文本短（~100字/条）。规则筛选（评分匹配+预算过滤+空间距离+多样性）在这个量级上的准确度不低于向量检索，且无 Embedding 调用开销。

## 4. 工具上下文总线（Tool Context Bus）

同一请求中工具间共享数据。Agent 不再需要手动在 prompt 中传递前一个工具的结果——工具执行后自动写入 `UserMemory._tool_context`，后续工具可通过 `get_tool_context()` 读取。同时请求结束时自动清空，不污染下次请求。

## 5. 为什么不用 LangGraph？

项目工具数量和流程复杂度尚不需要 DAG 编排。LangGraph 的额外学习成本和依赖引入与收益不匹配。如果后续需要多分支条件路由（如根据用户角色走不同工具链），可以考虑迁移。
